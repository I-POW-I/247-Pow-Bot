Build snapshot
What 24/7 POW Bot already does

The bot is already operating as a voice uptime tool, moderation helper, and server utility suite with a branded control panel and persistent state.

  • Maintains a 24/7 voice connection with silent audio and auto-recovery.
  • Posts a live control panel with buttons and member lookup tools.
  • Tracks voice sessions, server stats, logs, and streamer alerts.
Current focus
What still needs attention

A few areas still need real-world validation before the bot can be treated as fully production-hardened.

  • Confirm that persisted files survive restarts on the hosting platform.
  • Make startup and external API failures easier to understand at a glance.
  • Verify the roadmap-facing experience remains polished and consistent.
Roadmap scope
What this page is tracking

This roadmap is not a feature wish list. It is a live picture of the current implementation, the blockers that matter, and the next best additions.

  • Implemented features that are already live in the workspace.
  • Known risks tied to deployment, APIs, and command registration.
  • Upcoming work that improves onboarding and polish without changing the bot’s core identity.

Roles and permission guide
A practical template for who can control the bot and what they should be able to do
▼

Use this as the default permissions model for a server. It keeps control in the right hands, prevents casual users from changing the bot, and makes the experience predictable for admins.

Role / GroupWho it isCan doCannot do
Server ownerThe guild ownerUse the bot when no bot-control role is set, recover access, manage setupShould not be blocked from emergency admin actions
AdministratorTrusted staff with Manage ServerPost the panel, configure logs, set verification, manage general setupShould not control Leave / Force Leave unless explicitly granted
Bot control roleCustom role chosen with /setbotroleUse Leave and Force Leave controls from the panelShould not grant broader admin powers by default
Verification roleThe role assigned by /verify setupUnlock the community area or access gate you want verified users to reachShould not be used for bot control or moderation
Regular membersEveryone else in the serverUse public info controls like My Info or Lookup when allowedShould not join, leave, force leave, or change bot settings
Bot-side permissions
Required for the bot to function
Template
  • Connect and Speak for the voice channel
  • Send Messages and Embed Links for panel and log channels
  • Manage Roles only if you want verification role assignment
  • View Audit Log for cleaner moderation logging
User control rules
Recommended restrictions
Limit by default
  • Only admins should be able to post or refresh the panel
  • Only the bot-control role or server owner should use Leave and Force Leave
  • Regular members should never be able to change log channels or verification setup
  • Keep utility actions visible, but keep configuration actions locked down
If you want the strictest setup, give panel/setup commands to admins only, use a dedicated bot-control role for voice controls, and let everyone else use only the public info buttons.

Live now
01🔊
24/7 POW Bot voice connection system
Joins a voice channel and maintains the connection with silent audio and recovery logic
Implemented
▼
  • Voice join, leave, and force leave flows are wired into the control panel and slash commands
  • Ghost detection and auto-rejoin are handled by the heartbeat system
  • A silent audio player keeps Discord from dropping the connection
  • The bot restores the last known channel after restart when the host keeps state intact
  • Connection state is tracked centrally so panel actions and startup recovery stay in sync
02🧠
Control panel and runtime status
Panel with controls for join, leave, refresh, user lookup, and profile views
Implemented
▼
  • The panel is posted with /panel and updates automatically
  • Buttons support Join, Leave, Force Leave, My Info, Lookup User, and Refresh
  • Presence updates advertise the current channel and uptime for the bot
  • The panel is branded for 24/7 POW Bot so admins immediately know what they are controlling
  • The same panel state is used by the join and leave flows to avoid duplicate or stale controls
03📊
Voice and member tracking
Tracks VC sessions, member activity, and server stats in SQLite
Implemented
▼
  • VC session timestamps are stored and used for per-user stats and streaks
  • The status command reports connection info, uptime, reconnect count, active VC count, and session totals
  • Member profile embeds show account age, voice state, roles, and tracked VC history
  • The database layer keeps session records separate from guild config so stats and settings do not collide
  • Join and leave tracking continues across restarts when data files remain available
04📝
Logging system
Voice logs, message delete/edit logs, and member join/leave logs are all wired up
Implemented
▼
  • Voice activity logs cover joins, leaves, channel moves, mutes, deafens, streams, and camera events
  • Message delete and edit logs can post to a configured channel
  • Member join and leave logs include account age, member count, and role information when available
  • Each log path is routed through guild-specific configuration so servers can opt into only the events they want
  • The logging style stays consistent with the rest of the bot instead of feeling like a separate add-on
05✅
Verification and access controls
Admins can post a verification button and assign a role on click
Implemented
▼
  • /verify setup posts a verification embed and button
  • The chosen role is saved per guild and assigned to users who click the button
  • The bot checks role hierarchy before assigning the role
  • Verification messaging is branded to 24/7 POW Bot instead of sounding like generic Discord boilerplate
06📺
Streamer alerts
Kick, Twitch, and YouTube live notifications are supported with per-guild subscriptions
Implemented
▼
  • Commands include /addstreamer, /streamers, and /removestreamer
  • Live alerts post to a configured channel and can ping a chosen role
  • Twitch needs credentials, YouTube needs an API key, and Kick can still be affected by hosting IP restrictions
  • Streamer subscriptions are stored per guild so different servers can maintain different alert lists
  • The setup flow validates platform links before saving anything to the database
07🧰
Administration and utility commands
The bot now ships with a practical set of commands for maintenance and moderation
Implemented
▼
  • /setlogchannel routes different log types to different channels
  • /setbotrole controls who can use Leave and Force Leave from the panel
  • /purge deletes batches of messages safely
  • /clearcommands re-registers slash commands and clears stale ones
  • /help gives a quick overview of the bot’s features
  • The command set is intentionally practical rather than sprawling, which keeps the bot easier to maintain
Needs attention
08💾
Persistent config and database storage
Make sure config and panel state survive redeploys on Discloud without surprises
Needs validation
▼

24/7 POW Bot stores guild configuration and SQLite data locally, but hosting environments can still reset or isolate those files depending on deployment conditions. This needs a real host-side verification pass.

  • Confirm the Discloud data directory is persistent across deploys and restarts
  • Verify the panel message ID survives a restart without needing to recreate the panel
  • Document the fallback path if the host wipes runtime data
  • Check which files are mounted as persistent storage versus temporary runtime files
  • Confirm the bot starts cleanly when config exists but the cached voice connection does not
09🔐
External API reliability
Twitch and YouTube checks are live, but they depend on correct credentials and provider-side access
Needs polish
▼
  • Graceful fallback messaging should be clearer when Twitch credentials are missing or rejected
  • YouTube and Twitch checks should degrade cleanly when APIs rate-limit or return errors
  • Kick-specific failures should be documented and surfaced to admins more clearly
  • Live alert checks should fail closed instead of spamming broken notifications
  • Admins should be able to tell whether a failure is credential-related, rate-limit related, or platform-related
10⚙️
Startup and command registration reliability
The bot now starts cleanly, but command registration and host restarts should be monitored in production
Needs polish
▼
  • Keep slash command registration stable across deploys and guild changes
  • Log any missing permissions or Discord-side registration issues more explicitly
  • Make the startup flow more resilient if one module or feature fails to initialize
  • Surface command registration problems in the logs before they become support issues
  • Keep the startup path predictable so the bot can recover after a restart without manual cleanup
Next up

This roadmap is organized around shipping cycles instead of a loose task pile. That keeps 24/7 POW Bot moving in small, shippable steps while still separating completed work from active risks and intentionally deferred work.
